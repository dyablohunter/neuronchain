Welcome to NeuronChain

NeuronChain is a browser-based blockchain using a block-lattice DAG with stake-weighted voting and two-factor face + PIN locked keys. Your private key is double-encrypted — both your face and a 4-digit PIN are required to unlock it. No seed phrases. The native currency is the Neuron Unit (UNIT).

Getting Started

  • Step 1: Go to the Node tab and click Start Node to connect to the relay network. Do this first — Transfer and Contracts tabs are disabled until the node is running.
  • Step 2: Go to the Accounts tab, choose a username (3–24 chars, lowercase alphanumeric + dot), scan your face, then set a 4-digit PIN. You receive 1,000,000 UNIT.
  • Step 3: Save the backup secret key shown after creation — a short Base58 code. This is your secondary recovery method if face + PIN fails.
  • Step 4: Send UNIT in the Transfer tab — enter a recipient username, amount, then confirm with your PIN.
  • Step 5: Check the Explorer tab to see your transactions syncing across the network.

Neuron Unit (UNIT)

  • 1,000,000 UNIT minted per new account at creation
  • 3 decimal precision (milli-UNIT) - smallest transferable unit is 0.001 UNIT
  • All operations are zero fee - transfers, contracts, everything
  • Inflationary - total supply grows as new accounts are created (limited to the total number of people)
  • Mainnet: 1 account per face (1 human = 1 account)
  • Testnet: up to 3 accounts per face (for testing)
  • Usernames are always lowercase (auto-converted)

Two-Factor Keys (Face + PIN)

Account keys are protected by two independent factors. Compromising either one alone reveals nothing.

Encryption layers (inner → outer):

  • Inner layer — face factor: Stored canonical descriptor → quantize → PBKDF2-SHA-512 (100K iterations) → AES-256-GCM
  • Outer layer — PIN factor: 4-digit PIN + 32-byte random salt → PBKDF2-SHA-512 (600K iterations, OWASP 2024) → AES-256-GCM

Face enrollment:

  • Liveness check: Slowly turn your head left and right — requires ≥30 px nose travel and 2 direction reversals to prevent photo/video replay
  • Face capture: 3 captures averaged into a stable 128-dimensional canonical descriptor (face-api.js, 99.38% accuracy on LFW)
  • Stored canonical: The averaged descriptor is PIN-encrypted and saved inside the key blob (encryptedCanonical). On recovery, the PIN decrypts this stored descriptor to derive the face key — deterministically, regardless of lighting or angle changes between sessions.

How recovery works reliably: The live face scan at recovery time is compared to the stored canonical as a biometric identity check (tolerance-based). The actual AES face key is derived from the stored canonical, not the live scan — so it always matches the enrollment key exactly.

PIN:

  • Set once at account creation; independent per account
  • Required for: send tokens, deploy contract, account recovery, updating PIN/face
  • Not required for: contract calls (may come from other accounts or external inputs)
  • 5-minute session cache — after entering PIN (at login, recovery, or a transfer), no re-prompt for quick consecutive actions. Cleared on page close.

Privacy:

  • Raw face descriptor is PIN-encrypted inside the blob — never stored in plaintext
  • Only a SHA-256 hash of the face map is stored on-chain (for duplicate detection)
  • No images or raw descriptors leave the browser

Storage and backup:

  • On-chain blob: stored on the libp2p network + IndexedDB — available from any device
  • On-chain anchor (linkedAnchor): SHA-256(encryptedKeys:faceMapHash:pub) ties the blob to the account, detects tampering on recovery
  • Local wallet: AES-256-GCM encrypted with a per-session key in localStorage — cleared when the session ends
  • Backup secret key: A compact Base58 code (~263 characters) shown once at creation — encodes both private keys; secondary recovery without camera or PIN

Quantum-Safe Cryptography

Every account carries a hybrid classical + post-quantum key pair:

  • ECDSA P-256 — classical block signing (always present)
  • ML-DSA-65 (CRYSTALS-Dilithium) — quantum-safe block signing (present on all new accounts)
  • ECDH P-256 — classical key agreement / content encryption
  • ML-KEM-768 (CRYSTALS-Kyber) — quantum-safe key encapsulation

Every block carries both an ECDSA signature and an ML-DSA-65 pqSignature. Verification checks both. Legacy accounts (pre-upgrade) remain fully compatible — the PQ signature field is optional.

PQ private keys are stored inside the face+PIN double-encrypted blob. PQ public keys are stored on-chain so any peer can verify without unlocking your account.

Wrong PIN Backoff

Wrong PIN attempts trigger an exponential lockout to prevent brute force attacks:

  • Attempts 1–3: No delay
  • Attempt 4: 30 seconds
  • Attempt 5: 2 minutes
  • Attempt 6: 8 minutes
  • Attempt 7: 32 minutes
  • Attempt 8: 2 hours
  • Attempt 9: 8 hours
  • Attempt 10+: 24 hours

The attempt counter is stored both in IndexedDB (local, fast) and inside the encrypted blob (face-key-encrypted — transfers to new devices automatically and cannot be cleared without face auth). When the threshold is crossed, a lockout notice is broadcast via GossipSub — other nodes on the network will also delay processing blocks from the locked account.

Account Recovery

  • FaceID + PIN (primary): Enter your username → PIN prompt → face scan (3-sample capture, same as enrollment) → encrypted keys decrypted with both factors. Works from any device. The linkedAnchor is verified against on-chain data to detect blob tampering before decryption. Face key is derived from the PIN-decrypted stored canonical — not from the live scan — so recovery is reliable regardless of session-to-session variation in lighting or angle.
  • Backup secret key: Click "Recover with Key Pair" and paste the Base58 backup code saved at creation. Restores signing and encryption keys without a camera or PIN. The account's PIN requirement is automatically restored from the key blob.

Block-Lattice DAG

Instead of one global chain, every account has its own chain of blocks:

  • open — first block, created with FaceID + PIN, mints 1,000,000 UNIT
  • send — transfer UNIT (deducts from sender) — PIN required
  • receive — accept UNIT from a confirmed send (credits recipient)
  • deploy — deploy a smart contract — PIN required
  • call — call a smart contract method (no PIN — may come from external inputs)
  • update — change PIN or re-enroll face; updates linkedAnchor + PQ keys on-chain — PIN required

Transfers require two blocks: a send on the sender's chain and a receive on the recipient's chain. The recipient must claim the receive in the Transfer tab (or it is auto-claimed if their node is online).

Optimistic Confirmation

Blocks are confirmed instantly - no voting needed for normal transactions:

  • A block passes local validation (valid SHA-256 hash, ECDSA signature, sufficient balance, no fork) → confirmed immediately
  • No waiting for peer votes - 99.9% of transactions confirm in <1ms
  • Voting only triggers when a fork is detected (two blocks sharing the same parent = double-spend attempt)
  • During conflicts: peers vote weighted by their verified on-chain UNIT balance. Votes include a balance proof (the voter's chain head hash) so each node can independently verify the voter's stake from its own ledger instead of trusting self-reported values. >2/3 stake wins, loser rejected
  • Vote signatures are verified both at the relay (spam filter) and at each receiving client (trust-independent verification)
  • Conflicts resolve within 10 seconds
  • Vote replay protection: duplicate signatures are rejected, votes older than 60 seconds are ignored

P2P Networking

libp2p handles all P2P communication using a decentralised, multi-transport architecture:

  • WebRTC: browser peers upgrade to direct connections after initial bootstrap
  • GossipSub router (relay): while peers are on circuit-relay connections, the relay subscribes to all neuronchain/* topics and routes messages between browsers. Once peers upgrade to direct WebRTC, the relay is off the data path.
  • Circuit relay v2: when direct WebRTC fails (NAT), connections are tunnelled through a relay peer
  • Kademlia DHT: peer discovery is decentralised - nodes find each other via a distributed hash table, not a central registry
  • GossipSub topics: blocks sharded across 4 synapse topics by hash(accountPub) % 4; votes, accounts, and inbox signals each get their own topic
  • IndexedDB persistence: each node stores the full chain locally - no relay needed to read your own data
  • Bootstrap: npm run dev starts Vite + relay on port 9090. Custom bootstrap: localStorage.setItem('neuronchain_bootstrap', JSON.stringify([...multiaddrs]))
  • Separate relay: npm run relay runs the relay standalone for production
  • Testing Tunnel: npm run tunnel creates an HTTPS tunnel URL for multi-device testing
  • Resync: on-demand when switching tabs, with a 60-second background fallback and 20-second re-publish cycle
  • Single-tab lock: one node per browser (prevents account farming)
  • Auto-receive: incoming sends to local accounts are claimed automatically when the SEND block arrives via GossipSub. If the recipient's node was offline or briefly disconnected, the pending receive can be claimed manually (Transfer → Pending Receives). The sender re-broadcasts every 20 seconds.
  • Signed inbox signals: senders sign transfer notifications with their ECDSA key; recipients verify before processing - prevents relay fabrication of fake notifications
  • Production scaling: run multiple independent community relay nodes for resilience

Security

  • Two-factor key protection — private keys are double-encrypted: AES-256-GCM(face key) → AES-256-GCM(PIN key). Both factors are required to unlock.
  • Quantum-safe hybrid signing — every block carries both an ECDSA P-256 signature and an ML-DSA-65 (Dilithium) post-quantum signature. Both are verified.
  • PIN brute-force protection — PBKDF2-SHA-512 at 600K iterations (~300ms/attempt) plus exponential backoff (up to 24h lockout after 10 wrong attempts).
  • Decentralised lockout — when PIN attempts exceed the threshold, a lockout notice is broadcast via GossipSub; other nodes delay processing blocks from locked accounts.
  • Face descriptor privacy — the raw 128-dim descriptor is PIN-encrypted locally; only its SHA-256 hash is stored publicly on-chain.
  • On-chain binding (linkedAnchor) — SHA-256(encryptedKeys:faceMapHash:pub) ties the face+PIN blob to the account; verified on recovery to detect blob tampering.
  • SHA-256 block hashing — all block hashes computed via Web Crypto API
  • ECDSA signatures — every block is signed by its author and verified on receipt
  • Client-side vote verification — all incoming votes are cryptographically verified client-side; a malicious relay cannot fabricate or alter votes
  • Balance proofs — votes include the voter's chain head hash; receiving nodes verify the voter's balance from their own ledger
  • Peer-side validation — all incoming blocks, votes, and accounts are verified before being accepted; rogue peers cannot inject unsigned data
  • Noise protocol — all libp2p connections are encrypted and authenticated at the transport layer
  • Signed accounts — account data is ECDSA-signed; peers reject unsigned or spoofed accounts
  • Signed inbox signals — transfer notifications are ECDSA-signed by the sender; recipients verify before processing
  • Sandboxed contracts — smart contracts execute in an isolated Web Worker with no DOM/fetch/storage access; terminated after 3 seconds
  • Balance overflow protection — all amounts validated against Number.isSafeInteger
  • Generation governance — mainnet resets require a signed message from a known operator key
  • Encrypted local wallet — keys in localStorage are AES-256-GCM encrypted with a per-session key; cleared when the session ends
  • XSS protection — all peer-supplied data is HTML-escaped before rendering

Networks

  • Testnet: Up to 3 accounts per face. Can be reset from the Chain tab (wipes everything, increments generation counter). Any client can trigger a reset. For development and testing.
  • Mainnet: 1 account per face (1 human = 1 account). Generation resets require a signed message from a known operator key - no single client can wipe the network. Production network.

Both networks mint 1,000,000 UNIT per account and have zero fees.

Smart Contracts

Write contracts in JavaScript. Contracts execute in an isolated Web Worker sandbox - no access to the page, network, or storage APIs. Your code has access to state, caller, and args.

Simple Token (ERC-20 style):

// Deploy with: init("MyToken", "MTK", 1000000)
function init(name, symbol, totalSupply) {
  if (state.initialized) return;
  state.name = name; state.symbol = symbol;
  state.totalSupply = totalSupply;
  state.balances = {};
  state.balances[caller] = totalSupply;
  state.initialized = true;
  return { name, symbol, totalSupply };
}
function transfer(to, amount) {
  if (!state.balances[caller] || state.balances[caller] < amount)
    return { error: 'insufficient balance' };
  state.balances[caller] -= amount;
  state.balances[to] = (state.balances[to] || 0) + amount;
  return { success: true };
}
function balanceOf(address) { return state.balances[address] || 0; }
function getInfo() { return { name: state.name, symbol: state.symbol, totalSupply: state.totalSupply }; }

NFT Contract (ERC-721 style) - with content CIDs:

// Deploy with: init("MyNFTs", "MNFT")
// Mint with:   mint("My artwork", "<helia-cid>")
function init(name, symbol) {
  if (state.initialized) return;
  state.name = name; state.symbol = symbol;
  state.tokens = {}; state.nextId = 1;
  state.initialized = true;
  return { name, symbol };
}
function mint(metadata, cid) {
  const id = state.nextId++;
  state.tokens[id] = { owner: caller, metadata, cid, mintedAt: Date.now() };
  return { tokenId: id, owner: caller };
}
function transfer(tokenId, to) {
  const t = state.tokens[tokenId];
  if (!t) return { error: 'not found' };
  if (t.owner !== caller) return { error: 'not owner' };
  t.owner = to;
  return { success: true, tokenId, newOwner: to };
}
function ownerOf(tokenId) { return state.tokens[tokenId]?.owner || null; }
function tokenInfo(tokenId) { return state.tokens[tokenId] || null; }
function myTokens() {
  return Object.entries(state.tokens)
    .filter(([,t]) => t.owner === caller)
    .map(([id, t]) => ({ tokenId: Number(id), ...t }));
}

Content Storage

NeuronChain is a full content network. Posts, images, video, audio, HTML, CSS, JS, and JSON are stored on NeuronChain using its own content layer — not on any central server, and not on the public IPFS network. Content is served by NeuronChain peers only, using content-addressing (CIDs) and Bitswap for parallel chunk transfer.

  • Content-addressed: every file is stored by its SHA-256 CID. Any peer serving wrong bytes is automatically rejected - tamper-evident by construction.
  • On-chain anchor: the CID is recorded in block.contentCid or smart contract state, creating an immutable provenance proof. The content hash is the receipt.
  • ECDSA-signed: every store action is signed by your account key regardless of visibility - public content is still authentically attributed to you.
  • Served by any peer that holds a copy via Bitswap (parallel chunk transfer, BitTorrent-style). More holders = faster delivery.
  • IndexedDB persistence: content stored locally survives relay downtime - readable offline.

Dynamic visibility - choose per content item:

  • Public: stored unencrypted - anyone with the CID can read it. Use for posts, profiles, public media.
  • Private: encrypted with AES-256-GCM before storage using your account's content key (derived via ECDH). Only you can decrypt it. Use for personal data, drafts, private media.

Retrieval auto-detects visibility: tries decryption first, falls back to public read. Supported types: JSON, image, video, audio, HTML, CSS, JS, and any binary file.

Storage Market

Built-in peer storage with UNIT incentives:

  • Providers publish a storage-offer block announcing capacity (MB) and price (milli-UNIT / MB / day)
  • Beneficiaries accept with a storage-deal block
  • A 24h timer auto-pays from beneficiary to provider via a standard send block
  • If the beneficiary has insufficient balance, content is automatically unpinned from the provider's node
  • Content is stored on the NeuronChain network - content-addressed and encrypted before storage
  • CIDs are referenced in block.contentCid or smart contract state

Technical Specs

  • Key pairs: ECDSA P-256 + ECDH P-256 (Web Crypto API - no external dependency)
  • Block hashing: SHA-256 (Web Crypto API)
  • Content encryption: AES-256-GCM, key derived via PBKDF2 from ECDH private key
  • Face key derivation: PBKDF2 (100K iterations) → AES-256-GCM
  • Face descriptor: 128 dimensions, quantized to 0.05 bins
  • Consensus: Optimistic confirmation, conflict-only stake-weighted voting (>2/3 threshold, 10s timeout)
  • P2P: libp2p - WebRTC (browser↔browser), WebSocket (browser↔relay), circuit relay v2 (NAT traversal), GossipSub (pubsub), Kademlia DHT (peer discovery)
  • Content storage: NeuronChain content layer (Helia/Bitswap internally) - content-addressed, encrypted, parallel chunk transfer across NeuronChain peers
  • Persistence: IndexedDB (blocks, accounts, keyblobs, contracts) - no relay dependency
  • Keep-alive: Wake Lock API + Web Worker heartbeat + silent audio (prevents tab suspension)
  • dApp API: NeuronChainAPI - stable facade for building apps on NeuronChain
