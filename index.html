Overview

NeuronChain is a browser-based blockchain using a block-lattice DAG with stake-weighted voting and two-factor face + PIN locked keys. Every account has its own chain; transfers, contracts, and storage actions all land as signed blocks on those chains. No seed phrases — your private key is double-encrypted by your face and a 4-digit PIN.

The native currency is the Neuron Unit (UNIT). The network runs peer-to-peer in the browser via libp2p (WebRTC + GossipSub + Kademlia DHT) and @sinclair/smoke (HTTP-over-WebRTC + IndexedDB) for content storage — no central server on the data path.

Quick Start

  1. Start Node — Node tab → Start Node. The node connects to the relay, joins GossipSub, and begins syncing. Transfer and Contracts tabs are disabled until this step.
  2. Create Account — Accounts tab → choose username (3–24 chars, lowercase alphanumeric + dot) → liveness check (turn head slowly) → face enrollment (3 captures) → set 4-digit PIN → 1,000,000 UNIT minted.
  3. Save Backup Key — copy the Base58 secret key shown after creation. This is your secondary recovery method without camera or PIN.
  4. Send UNIT — Transfer tab → recipient username → amount → confirm with PIN.
  5. Explore — Explorer tab shows your transaction history syncing across the network in real time.
Multi-device: run npm run tunnel in a second terminal to get an HTTPS tunnel URL. Open it on your phone to test cross-device sync.

UNIT Currency

  • 1,000,000 UNIT minted per new account at creation (FaceID mandatory)
  • 3 decimal precision — smallest unit is 0.001 UNIT (1 milli-UNIT)
  • Zero fees — transfers, contracts, storage, everything
  • Inflationary — supply grows with each new account, capped by the number of living humans
  • Mainnet: 1 account per face  |  Testnet: up to 3 per face
  • Usernames are always lowercase (auto-converted)

Storage providers also earn UNIT via daily storage-reward blocks — new UNIT minted proportional to stored GB × uptime factor.

Block Types

Every account has its own chain of signed blocks. Each block references the hash of its parent, forming a tamper-evident chain.

Transfers require two blocks: a send on the sender's chain and a receive on the recipient's chain. Receives are auto-claimed when the recipient's node is online. The sender re-broadcasts every 20 seconds so receives are claimed on reconnect if the node was temporarily offline.

Two-Factor Keys (Face + PIN)

Private keys are protected by two independent cryptographic factors. Compromising one factor alone reveals nothing.

Encryption layers (inner → outer)

  1. Face factor (inner): stored canonical face descriptor → quantize to 0.05 bins → PBKDF2-SHA-512 (100K iterations) → AES-256-GCM key
  2. PIN factor (outer): 4-digit PIN + 32-byte random salt → PBKDF2-SHA-512 (600K iterations, OWASP 2024) → AES-256-GCM key

Face enrollment

  • Liveness check: slowly turn head left and right — requires ≥ 30 px nose travel and 2 direction reversals to prevent photo/video replay
  • 3-sample average: 3 captures averaged into a stable 128-dimensional canonical descriptor (face-api.js, 99.38% accuracy on LFW)
  • Stored canonical: the averaged descriptor is PIN-encrypted and saved in the blob as encryptedCanonical

Why recovery is reliable across devices

Face descriptors vary between sessions (lighting, angle). Key derivation uses the stored canonical — not the live scan — so the face key is always identical to the one used at enrollment.

  1. At enrollment: averaged canonical is PIN-encrypted → stored as encryptedCanonical
  2. At recovery: PIN entered first → PIN key decrypts encryptedCanonical → stored canonical recovered → face key derived deterministically
  3. Live scan used only as a biometric identity check (tolerance-based distance comparison), not for key derivation

PIN

  • Set once at creation; independent per account
  • Required for: send, deploy, open, recovery, update PIN/face
  • Not required for: contract calls (may originate from other accounts)
  • 5-minute session cache — no re-prompt for quick consecutive actions; cleared on page close

On-chain binding

linkedAnchor = SHA-256(encryptedKeys + ":" + faceMapHash + ":" + pub) — stored in the open block, updated on every update block. Verified on recovery to detect blob tampering.

Account Recovery

FaceID + PIN (primary)

Enter username → PIN prompt → face scan (3-sample capture) → keys decrypted with both factors. Works from any device. The linkedAnchor is verified against on-chain data before decryption. Face key is derived from the stored canonical — not the live scan — so recovery is reliable regardless of lighting or angle changes.

Backup secret key

Click Recover with Key Pair and paste the Base58 code saved at creation (~263 characters). Restores signing and encryption keys without a camera or PIN. The account's PIN requirement is restored from the blob automatically.

Re-enrolling your face

Account tab → Update PIN / Face → Update Face. Identity is verified by PIN + liveness check before the camera opens. The keys already loaded in the active session are re-encrypted under the new face key — no blob decryption required. Works correctly after a PIN change with no extra steps.

Save your backup key. It is shown only once at account creation. If you forget your PIN and lose the backup key, account recovery is not possible.

Optimistic Consensus

Blocks are confirmed instantly — no voting needed for normal transactions. Voting only triggers when a fork is detected (two blocks sharing the same parent = double-spend attempt).

Normal path (> 99.9% of blocks)

  1. Block passes local validation: valid SHA-256 hash, ECDSA + ML-DSA-65 signature, sufficient balance, no fork
  2. Block confirmed immediately — < 1 ms latency

Conflict path (fork detected)

  1. Network detects two blocks with the same previousHash
  2. Nodes broadcast ECDSA-signed votes including chainHeadHash as a balance proof
  3. Each node verifies the voter's stake from its own ledger (no trust in self-reported values)
  4. First side to reach > 2/3 of total stake wins (10-second timeout)
  5. Rejected chain rolled back; unclaimed sends restored
Vote replay protection: duplicate signatures are rejected; votes older than 60 seconds are ignored. All vote signatures are verified client-side — a malicious relay cannot fabricate or alter votes.

Content Storage

NeuronChain is a full content network — posts, images, video, audio, HTML, CSS, JS, and JSON stored on NeuronChain using its own content layer. No central server; content is served peer-to-peer via HTTP over WebRTC (@sinclair/smoke).

Properties

  • Content-addressed: every file stored by its SHA-256 CID (bafkrei…). SHA-256(fetched bytes) is verified against the CID before accepting — tamper-evident by construction.
  • On-chain anchor: the CID is recorded in block.contentCid or smart contract state, creating an immutable provenance proof.
  • ECDSA-signed: every store action is signed by the account key — public content is authentically attributed to you.
  • Served by any peer that holds a copy via smoke Http on virtual port 5891. Known holders from storage receipts are tried first before broadcasting to all peers.
  • IndexedDB persistence: content stored locally survives relay downtime — readable offline.
  • WebRTC signaling via GossipSub: smoke peer connections are coordinated through the GossipSub mesh. The relay's /smoke-hub is fallback only — relay is not on the content data path.

Decentralised Storage Ledger

Built-in automated storage incentives — no marketplace, no manual deals.

Provider lifecycle

  1. Register — publish a storage-register block with capacity in GB. Automatic heartbeats and daily reward blocks begin.
  2. Heartbeat — a storage-heartbeat block is issued every ~4 hours. Each heartbeat records the smoke Hub address on-chain so peers survive restarts.
  3. Reward — a self-issued storage-reward block mints UNIT once per 24 h. The network verifies the amount against on-chain heartbeat count — over-claiming is rejected. A chain scan prevents double-issue from multiple devices.

Earning rate

daily_reward = 1 UNIT × capacityGB × uptime_factor × latency_factor × spotcheck_factor

uptime_factor    = heartbeat_blocks_today / 6        (max 1.0 — one every ~4 h)
latency_factor   = 1000ms / avg_retrieval_latency    (max 1.0; response rank weighted)
spotcheck_factor = spot_checks_passed / spot_checks_received

Response rank (arrival order among all providers for a CID) is factored into the latency score — providers who respond first earn more.

Content distribution

  • Uploaders select up to 10 providers by weighted-random (weight = capacityGB × score)
  • A signed CacheRequest is broadcast via GossipSub with the uploader's smoke address
  • Providers fetch each block via smoke Http (HTTP-over-WebRTC) and store in IndexedDB
  • If confirmed providers for a CID fall below half the redundancy target (5), StorageManager automatically redistributes to fill the gap

Spot checks & integrity

Nodes periodically fetch blocks directly from each provider's smoke address and verify SHA-256(bytes) === CID. Providers serving wrong bytes are removed from the confirmed-provider set and excluded from future selection.

Content deletion

The owner broadcasts a signed DeleteRequest via GossipSub. Any node that holds a copy verifies the ECDSA signature and immediately drops the block from IndexedDB. Ownership is enforced per-CID — only the original uploader's public key can delete.

Smart Contracts

Contracts are written in plain JavaScript and execute in an isolated Web Worker sandbox — no access to DOM, fetch, localStorage, WebSocket, or any browser API. Execution is terminated after 3 seconds. Load templates from the Contracts tab, edit the code, and click Deploy.

Sandbox Environment

Each contract runs in its own Web Worker context with three globals injected at call time:

GlobalTypeDescription
stateobjectPersistent JSON object that survives across calls. Read and write freely — changes are committed after each successful call. Starts as {} on first deploy.
callerstringThe ECDSA public key (hex) of the account that created the call block. Use this for ownership checks and per-user balances.
argsany[]Array of arguments passed to the method by the caller. Types are preserved as JSON.

State & Variables

  • state persists in IndexedDB across calls and page reloads — treat it as your database
  • Any JSON-serializable value can be stored: strings, numbers, booleans, arrays, nested objects
  • Only named top-level function declarations are callable — arrow functions and methods on objects are not exposed
  • Return any JSON-serializable value from a function — it is stored in the call block's result field and returned to the caller
  • Return { error: '...' } to signal failure — state is only committed on success, so an error return leaves state unchanged
  • No async/await — all logic must be synchronous
Contract calls on conflicted blocks are deferred until the conflict resolves. Always guard initialisation with if (state.initialized) return to prevent re-init after a fork rollback.

Token Contract — Full Example

A fungible token with owner-controlled minting, peer-to-peer transfers, and balance queries. Load from Contracts tab → Simple Token template, then call init after deploying.

// ── STATE SHAPE ────────────────────────────────────────────────
// state.initialized  boolean
// state.owner        string  (deployer pub key — only they can issue)
// state.name         string
// state.symbol       string
// state.totalSupply  number
// state.balances     { [address: string]: number }

// Step 1 — call once: init("MyToken", "MTK", 1000000)
function init(name, symbol, totalSupply) {
  if (state.initialized) return { error: 'already initialized' };
  state.name        = name;
  state.symbol      = symbol;
  state.totalSupply = totalSupply;
  state.owner       = caller;          // deployer becomes owner
  state.balances    = {};
  state.balances[caller] = totalSupply; // full supply to deployer
  state.initialized = true;
  return { name, symbol, totalSupply };
}

// Mint additional tokens to any address (owner only)
// Call: issue("<address>", 500000)
function issue(to, amount) {
  if (caller !== state.owner) return { error: 'not authorized' };
  if (!Number.isInteger(amount) || amount <= 0)
    return { error: 'invalid amount' };
  state.balances[to]  = (state.balances[to] || 0) + amount;
  state.totalSupply  += amount;
  return { success: true, to, amount, newTotal: state.totalSupply };
}

// Transfer between accounts
// Call: transfer("<recipient-address>", 100)
function transfer(to, amount) {
  const balance = state.balances[caller] || 0;
  if (balance < amount) return { error: 'insufficient balance' };
  state.balances[caller] = balance - amount;
  state.balances[to]     = (state.balances[to] || 0) + amount;
  return { success: true, from: caller, to, amount };
}

// Queries — callable by anyone
function balanceOf(address) { return state.balances[address] || 0; }

function getInfo() {
  return { name: state.name, symbol: state.symbol, totalSupply: state.totalSupply };
}

function allBalances() { return state.balances; }

Workflow

  1. Deploy — Contracts tab → paste code → Deploy → contract gets a unique ID (its block hash)
  2. Init — Call → method: init → args: "MyToken", "MTK", 1000000
  3. Issue more tokens — Call → method: issue → args: "<address>", 500000 (owner only)
  4. Transfer — Call → method: transfer → args: "<address>", 100
  5. Check balance — Call → method: balanceOf → args: "<address>"

NFT Contract — Full Example

A non-fungible token collection where each token carries a content CID pointing to media stored on NeuronChain's content layer. Upload media via the Storage tab to get a CID, then pass it to mint.

// ── STATE SHAPE ────────────────────────────────────────────────
// state.initialized  boolean
// state.owner        string  (collection creator)
// state.name, state.symbol  string
// state.nextId       number  (auto-incrementing)
// state.tokens       { [id]: { owner, metadata, cid, mintedAt } }

// Step 1 — call once: init("My Art", "MART")
function init(name, symbol) {
  if (state.initialized) return { error: 'already initialized' };
  state.name = name; state.symbol = symbol;
  state.owner = caller; state.nextId = 1; state.tokens = {};
  state.initialized = true;
  return { name, symbol };
}

// Mint a token — cid = SHA-256 CID from the Storage tab
// Call: mint("<recipient>", "Artwork #1", "bafkrei...")
function mint(to, metadata, cid) {
  if (caller !== state.owner) return { error: 'not authorized' };
  const id = state.nextId++;
  state.tokens[id] = { owner: to, metadata, cid, mintedAt: Date.now() };
  return { tokenId: id, owner: to, cid };
}

// Transfer token ownership
// Call: transfer(1, "<new-owner>")
function transfer(tokenId, to) {
  const token = state.tokens[tokenId];
  if (!token)               return { error: 'token not found' };
  if (token.owner !== caller) return { error: 'not owner' };
  token.owner = to;
  return { success: true, tokenId, newOwner: to };
}

// Queries
function ownerOf(tokenId) { return state.tokens[tokenId]?.owner ?? null; }
function tokenInfo(tokenId) { return state.tokens[tokenId] ?? null; }

// Returns all tokens owned by the caller
function myTokens() {
  return Object.entries(state.tokens)
    .filter(([, t]) => t.owner === caller)
    .map(([id, t]) => ({ tokenId: Number(id), ...t }));
}

function allTokens() {
  return Object.entries(state.tokens)
    .map(([id, t]) => ({ tokenId: Number(id), ...t }));
}

Workflow with Content API

  1. Upload media — Storage tab → upload image → get CID (e.g. bafkreiabc…)
  2. Distribute — Storage tab → Distribute → sends the CID to up to 10 providers
  3. Deploy contract — Contracts tab → paste code → Deploy
  4. Init collection — Call → method: init → args: "My Art", "MART"
  5. Mint NFT — Call → method: mint → args: "<owner-address>", "Artwork #1", "bafkreiabc…"
  6. Transfer NFT — Call (as token owner) → method: transfer → args: 1, "<new-owner>"
  7. View collection — Call → method: allTokens → returns all tokens with their CIDs
The cid field in each token is a permanent content address — any peer holding the media can serve it via smoke Http. Token ownership and media ownership are separate: the contract records who owns the NFT; the CID points to the content bytes.

Content API

The NeuronChainAPI facade in src/api/neuronchain-api.ts provides a stable interface for building apps on NeuronChain. Instantiate with a running NeuronNode:

import { NeuronChainAPI } from './src/api/neuronchain-api';
const api = new NeuronChainAPI(node); // node = running NeuronNode instance

Storing Content

Public content (unencrypted)

// Store a JSON object
const encoder = new TextEncoder();
const { cid } = await api.storeContentPublic(
  encoder.encode(JSON.stringify({ title: 'Hello', body: '...' })),
  'application/json',
  'my-post'        // optional filename hint
);
// cid: 'bafkrei...' — permanent, immutable content address

// Store a file from <input type="file">
const file  = fileInput.files[0];
const bytes = new Uint8Array(await file.arrayBuffer());
const { cid: imgCid } = await api.storeContentPublic(bytes, file.type, file.name);

Private content (AES-256-GCM encrypted)

// myKeys = { pub, priv, epub, epriv, pqPub?, pqPriv?, ... }
const { cid: privateCid } = await api.storeContent(
  imageBytes,
  'image/jpeg',
  myKeys,
  'profile-photo'
);
// Encrypted before touching IndexedDB or any peer — only your keys can decrypt it

With metadata

// Returns cid + ContentMeta { name, mimeType, size, createdAt, contentCid }
const { cid, meta } = await api.store.storeWithMetaPublic(bytes, 'image/png', 'avatar');
console.log(meta.size, meta.createdAt, meta.contentCid);

Retrieving Content

// Public
const bytes = await api.retrieveContentPublic(cid);
const text  = new TextDecoder().decode(bytes);

// Private (decrypts with your keys)
const bytes = await api.retrieveContent(privateCid, myKeys);

// Auto-detect — tries decryption first, falls back to public
const bytes = await api.retrieveContentAuto(cid, myKeys);

// With metadata
const { data, meta } = await api.store.retrieveWithMetaPublic(cid);
console.log(meta.mimeType, meta.name, data.byteLength);

Retrieval checks local IndexedDB first (instant if cached), then queries known providers from receipts (targeted fetch), and finally broadcasts to all peer fallbacks. Every fetch verifies SHA-256(bytes) === CID before returning.

Distributing & Deleting Content

// Push CIDs to up to 10 providers (weighted-random by score × capacity)
await api.distributeContent([cid, imgCid], myPub, myKeys);

// Delete locally + broadcast signed delete to all known holders
await api.deleteContent([cid], myPub, myKeys);
// Holders verify your signature — only you can delete your content

Complete end-to-end workflow

// 1. Upload media
const { cid } = await api.storeContentPublic(bytes, 'image/jpeg', 'cover');

// 2. Anchor on-chain by passing the CID to a contract call
await api.callContract(contractId, 'mint', [myPub, 'Cover art', cid], myPub, myKeys);

// 3. Distribute to providers for redundancy
await api.distributeContent([cid], myPub, myKeys);

// 4. Any peer can retrieve it
const data = await api.retrieveContentPublic(cid);

// 5. Delete when no longer needed
await api.deleteContent([cid], myPub, myKeys);

Quantum-Safe Cryptography

Every account carries a hybrid classical + post-quantum key pair. All four private keys live inside the face+PIN double-encrypted blob; public keys are stored on-chain.

KeyAlgorithmPurpose
pub / privECDSA P-256Block signing (classical, always present)
epub / eprivECDH P-256Key agreement / content encryption
pqPub / pqPrivML-DSA-65 (Dilithium)Block signing (quantum-safe, FIPS 204)
pqKemPub / pqKemPrivML-KEM-768 (Kyber)Key encapsulation (quantum-safe, FIPS 203)

Every block carries both an ECDSA signature and an ML-DSA-65 pqSignature. Both are verified on receipt. Legacy accounts without PQ keys are fully backward-compatible — the PQ signature field is optional.

Implemented in pure TypeScript via @noble/post-quantum — no WASM required.

PIN Backoff & Lockout

Wrong PIN attempts trigger an exponential delay: delay_s = attempt > 3 ? Math.min(86400, 30 × 4^(attempt−3)) : 0
